# ExpenseMind

## Архитектура

Frontend → Go Backend → DB
→ ML (HTTP)

## Инварианты

* Go — единственная точка входа
* Вся бизнес-логика в Go
* ML сервис — stateless (только вычисления)
* ML:

  * не имеет доступа к БД
  * не содержит бизнес-логики
  * не парсит файлы
* Парсинг PDF/CSV только в Go
* Взаимодействие: HTTP + JSON

## Зависимости

* handler → service → repository
* service → parser, ml
* зависимости направлены только вниз

## Финансовая модель

* transactions = фактические операции (прошлое и настоящее)
* recurring_rules = правила регулярных операций (будущее)
* будущее не хранится в БД, а генерируется в service

## Работа с датой

* service обязан учитывать текущую дату (today)
* прошлое (<= today) берётся только из transactions
* будущее (> today) генерируется из recurring_rules

## Принципы recurring

* хранит правила, а не фактические операции
* используется только для генерации будущих значений
* изменения правил не должны ломать прошлые данные

## Потоки (инварианты)

* Upload:

  * parser → repository
  * ML НЕ вызывается

* Forecast:

  * асинхронный (job + worker)
  * ML вызывается только в фоне

## Правила

* соблюдать API контракты
* не усложнять решения
* использовать только релевантные части проекта
