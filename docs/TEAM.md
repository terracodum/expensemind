# ExpenseMind — Объяснение проекта для команды

Читай это если хочешь понять проект целиком — что это, зачем, как работает и почему именно так.

---

## Что это такое?

ExpenseMind — локальный инструмент для учёта личных финансов студента.

Пользователь устанавливает его на свой компьютер, запускает одной командой и работает через браузер. Все данные хранятся локально — ничего не уходит в интернет.

Грубо говоря: это как банковское приложение, но оно живёт у тебя на компьютере и ты сам себе сервер.

---

## Зачем это нужно?

Студенты тратят деньги хаотично и не понимают куда уходит стипендия. Существующие приложения требуют регистрации и загрузки личных финансовых данных на чужие сервера — многие этого не хотят.

ExpenseMind решает это так:
- данные не покидают компьютер пользователя
- есть аналитика по категориям расходов
- ML модель прогнозирует баланс на следующий месяц
- можно загрузить выписку из Т-Банка (PDF) вместо ручного ввода

---

## Как пользователь работает с приложением?

```
1. Установил Docker Desktop (один раз)
2. Написал в терминале: expensemind
3. Открылся браузер на localhost:3000
4. Загрузил PDF выписку из Т-Банка
5. Видит графики расходов по категориям
6. Видит прогноз баланса на 30 дней вперёд
```

Всё. Никакой регистрации, никаких серверов, никакой утечки данных.

---

## Из чего состоит проект?

Проект состоит из трёх независимых сервисов которые общаются между собой:

```
Браузер (React)
      ↓
Go Backend  ←→  База данных (SQLite)
      ↓
Python ML сервис
```

**React (фронтенд)** — то что видит пользователь в браузере. Графики, таблицы, кнопка загрузки PDF. Сам ничего не считает — только показывает данные которые отдаёт бэк.

**Go Backend** — мозг системы. Принимает запросы от браузера, парсит PDF, сохраняет транзакции в базу, считает аналитику, вызывает ML сервис. Вся бизнес-логика живёт здесь и только здесь.

**Python ML сервис** — калькулятор для прогнозов. Получает от Go готовые данные, считает прогноз баланса, возвращает результат. Больше ничего не делает — не знает про базу данных, не знает про пользователя.

**SQLite** — база данных. Просто хранит транзакции на диске пользователя.

---

## Почему локально, а не сайт в интернете?

Потому что никто не хочет загружать свои финансовые данные на чужой сервер.

При этом технически это полноценный веб-сервис — HTTP, REST API, браузерный интерфейс. Просто сервер запущен на компьютере самого пользователя. Это нормальная практика — так же работают Gitea, Obsidian, Portainer и другие self-hosted инструменты.

На защите это звучит так: *"Мы разработали веб-сервис который разворачивается локально, что обеспечивает полную приватность финансовых данных пользователя."*

---

## Почему Go для бэка, а не Python или Node?

- Go компилируется в один бинарник — быстрый старт, маленький Docker образ
- Go отлично подходит для HTTP серверов — это его основное применение
- Горутины позволяют обрабатывать много запросов параллельно без боли
- Строгая типизация — меньше ошибок в рантайме

---

## Почему Python отдельным сервисом, а не внутри Go?

Потому что ML библиотеки (pandas, scikit-learn) существуют только в Python. Вместо того чтобы тащить Python в Go монолит — выносим ML в отдельный микросервис. Go вызывает его по HTTP и получает готовый результат.

Плюс это разделение ответственности — ML разработчик работает независимо от бэк разработчика.

---

## Главные правила проекта

**Правило 1 — Go бэк это единственная точка входа**

Браузер общается только с Go. Python ML сервис не доступен извне — только Go может к нему обращаться.

**Правило 2 — ML сервис не знает про базу данных**

Python сервис получает готовые данные от Go и возвращает результат. Он не подключается к SQLite, не читает файлы, не хранит состояние между запросами.

**Правило 3 — вся бизнес-логика в Go**

Что считать расходом, как категоризировать транзакции, как агрегировать данные для ML — всё это решает Go. Python только считает прогноз по готовым данным.

**Правило 4 — PDF парсится только в Go**

Пользователь загружает PDF выписку Т-Банка. Go парсит её, извлекает транзакции, сохраняет в базу. PDF после этого удаляется. Python про PDF вообще не знает.

## Что делает каждый участник команды?

**Бэк (Go) + фронт (React) + архитектура:**
- REST API
- PDF парсер выписок Т-Банка
- бизнес-логика и аналитика
- интеграция с ML сервисом
- React интерфейс
- Docker конфигурация

**ML сервис (Python):**
- один эндпоинт `POST /internal/v1/predict`
- принимает временной ряд баланса + фичи
- возвращает прогноз на N дней вперёд
- полное ТЗ в файле `docs/ml_spec.md` — там всё написано

**Документация для курсовой:**
- описание архитектуры
- обоснование технических решений
- всё что нужно есть в `README.md` и `CLAUDE.md`
